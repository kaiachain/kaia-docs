# バリデータ・オンボーディング

マネージャーアカウントの設定が完了したら、バリデータとして登録するためにオンチェーン登録の手順を実行することができます。 このプロセスを開始する前に、まず実際のコンセンサス・ノード（kcn）を実行する必要があることに注意してください。

カイアのバリデーターとして参加するには、1つのステーキング契約を展開し、ステーキング契約の住所とその他の必要情報を提出する必要があります。

:::info パーミッションレス・フェーズ1／フェーズ2

パーミッションレスフェーズ1では、スムーズな運用のため、カイアチームによる**管理承認**が必要となります。 新しいバリデーターオペレーターはオンボーディングリクエストを提出することができ、カイアチームは内部承認プロセスを経てオンチェーン登録を完了する。

今後予定されている**パーミッションレス・フェーズ2**からは、カイア・チームの管理承認なしに情報を直接オンチェーン登録できるようになります。

:::

登録後、バリデータの情報は**AddressBook**と**SimpleBlsRegistry**コントラクトに記録され、Kaiaノードはコンセンサス時に参照する。 Self Validator Registrationの技術的な詳細については、[KIP-277](https://kips.kaia.io/KIPs/kip-277) を参照のこと。

## 最初のステーキング契約を展開する<a id="deploy-your-first-staking-contract"></a>

Deploy Staking Contract\*\*メニューに移動し、ステーキング契約をデプロイします。

現在、1つのステーキングコントラクトに複数の管理者アカウントを登録することができますが、**パーミッションレスフェーズ2**からは、1つのステーキングコントラクトにつき1つの管理者アカウントのみが許可されます。 このため、カイアセーフのアカウントを1つだけ管理者として登録することをお勧めします。

ステーキング・コントラクトをデプロイする前に、デプロイを実行するために、管理者の上に追加の一時アカウントが必要です。 このアカウントは**契約バリデーター**と呼ばれ、後方互換性のために存在する。 通常、ステーキング契約管理者アカウントとバリデータ管理者アカウントは別々に管理されるため、 バリデータ管理者アカウントをバリデータとして再利用することができる。

ステーキングコントラクト管理ウォレットとテンポラリーコントラクトバリデータウォレットは、それぞれ少量のKAIAを保持していなければならない。 すべての準備が整ったら、以下のステップに進む。

![杭打ち契約書の展開](/img/nodes/become-a-validator/image07.png)

1. カイアネットワークにオンボードしていますか]\*\*チェックボックスをクリックします。

2. 事前に用意した契約バリデーターのアカウントアドレスを入力します。

3. コンセンサスノードID\*\*には、CNコンソールで`admin.nodeInfo.nodeAddress` RPCを呼び出して返されるアドレスを入力する。

4. 報酬アドレス\*\*は、パブリック委任を使用するかどうかによって異なります。
   - パブリック・デレゲーションなしでバリデーターとして参加するには、報酬を直接受け取るアドレスを入力してください。
   - パブリック委任を\*\*有効にしてオンボードするには、**パブリック委任**チェックボックスをクリックします。 その後、公的委任契約を通じて、報酬が自動的に代議員に分配される。

![公的委任セクション](/img/nodes/become-a-validator/image08.png)
![公的委任セクション（続き）](/img/nodes/become-a-validator/image09.png)

5. ステーキング契約の管理者アドレスとマルチシグの閾値を入力します。 管理者アドレス**にカイアセーフのウォレットアドレスを入力し、閾値を「1」に設定すると、マルチシグ機能はカイアセーフが処理します。 複数の管理者アドレスを入力し、閾値を任意の値に設定した場合、マルチシグ機能は**Manage Staking\*\*メニュー内で処理されます。

![デプロイ契約ボタン](/img/nodes/become-a-validator/image10.png)

6. Deploy Contract]**をクリックしてトランザクションを実行する。 コントラクトがデプロイされると、**[初期化されていない]\*\*状態になります。

## ステーキング契約を初期化する<a id="initialize-the-staking-contract"></a>

新しく配置されたステーキング・コントラクトは、使用する前に初期化されなければならない。 必要な情報を入力し、各管理者アカウントから1回トランザクションを送信してウォレットを確認すると、契約が使用可能になります。

セット・ステーキング・トラッカー](/img/nodes/become-a-validator/image11.png)

1. Set Staking Tracker]\*\*をクリックし、ステーキング契約書にステーキングトラッカーのアドレスを書き込みます。 正しいステーキング・トラッカーのアドレスが自動的に入力されます。

GCのIDを設定](/img/nodes/become-a-validator/image12.png)

2. カイアチームからGC IDを受け取り、入力する。 Set GC ID]**をクリックすると、GC IDがステーキング契約に書き込まれます。 Permissionless Phase 2**より、GC IDは自動的に割り当てられます。

![Public Delegation info](/img/nodes/become-a-validator/image13.png)
![Public Delegation info (cont)](/img/nodes/become-a-validator/image14.png)

3. Public Delegationが有効になっている場合は、関連情報を入力する。 契約展開時にパブリック委任が有効になっていない場合は、この手順をスキップします。
   1. **オーナー**：コミッションの受取人とコミッションレートを変更できるアカウント。
   2. **Commission recipient**: コミッションを受け取るアカウント。
   3. **手数料率**：ベーシス・ポイントで「0」から「10000」の間の値。
   4. **GC名**：pdKAIAトークン名として公開される短い名前。 例えば、GC 名が `Hello` である場合、Public Delegation のデポジットトークン名は `Hello-pdKAIA` となる。 (例[kaiascan search for pdKAIA tokens](https://kaiascan.io/search?tabId=tokens&keyword=pdkaia&page=1))

レビュー条件](/img/nodes/become-a-validator/image15.png)

4. 先に設定したコントラクトバリデーターと、すべてのステーキングコントラクト管理者からそれぞれ1回ずつトランザクションを送信し、ウォレットを検証する。 それぞれのウォレットを順番に使用してログインし、\*\*[審査条件]\*\*をそれぞれ1回ずつクリックします。

![Deposit & Init (1)](/img/nodes/become-a-validator/image16.png)
![Deposit & Init (2)](/img/nodes/become-a-validator/image17.png)

5. 最後に、\*\*[Deposit & Init]\*\*をクリックして、契約の初期化を完了する。

## オンボーディング・リクエストを提出する<a id="submit-an-onboarding-request"></a>

ホーム**メニューに戻ると、配置された杭打ち契約は**[初期化]\*\*状態で表示されます。

初期化後のホーム](/img/nodes/become-a-validator/image18.png)

オンボードバリデーター]\*\*をクリックし、カイアチームにオンボードリクエストを送信します。

バリデーター搭載](/img/nodes/become-a-validator/image19.png)

オンボーディング申請フォーム](/img/nodes/become-a-validator/image20.png)

1. 表示された情報を確認する。
2. ノードの情報を確認する。 コンセンサス・ノードID**アドレスは、少なくとも**10 KAIA\*\*（[ガス抽出](../../../build/tutorials/ga-tutorial/ga-intro.md)および[MEVオークション](../../../build/tutorials/mev-auction-sdk-guide.md)トランザクションの処理に必要な最小値）を保持していなければならない。 10KAIAは差し引かれることはなく、これらのトランザクションの処理中に一時的に使用され、その後すぐに返却される。

BLS公開鍵情報](/img/nodes/become-a-validator/image21.png)

3. ノードにBLS公開鍵情報を問い合わせ、入力する。 CN コンソールで、`admin.nodeInfo.blsPublicKeyInfo` RPC を呼び出し、返された `publicKey` と `pop` の値を入力する。

オンボーディング・リクエストの提出](/img/nodes/become-a-validator/image22.png)

4. オンボーディングリクエストを送信]\*\*をクリックして、オンボーディングリクエストを送信します。
